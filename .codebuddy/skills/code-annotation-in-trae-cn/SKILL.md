# Code Annotation in Trae CN Skill

## 概述

当用户输入 `/code-annotation-in-trae-cn` 指令时触发本 skill。负责评估"学生"（受训练的大模型）在上一轮 prompt 中的产出效果，按固定格式在 `.history/` 目录下生成评估文件。

## 工作流程

### Step 1: 收集评估素材

1. **读取上一轮 prompt**：读取 `.prompt/previous_prompt.md`，获取上一轮向"学生"提问的 prompt 具体内容。
2. **读取学生的文字回复**：读取 `.prompt/previous_feedback.md`，获取"学生"的文字回复内容。
3. **获取 git 暂存区代码变更**：通过 `git diff --cached` 获取 git 暂存区中的代码变更。如果暂存区为空，则提示用户手动维护。

### Step 2: 确定文件名

文件名格式：`TASK_` + 日期（YYMMDD） + `_` + 序号 + `.md`

- **日期**：使用系统当前日期。必须自动获取，不可硬编码。
- **序号**：检查 `.history/` 目录下已有哪些以 `TASK_` + 当天日期 开头的文件，序号取最大值 + 1。
  - 例：今天已有 `TASK_260511_01.md`，则新文件为 `TASK_260511_02.md`。
  - 如果没有当天文件，则序号为 `01`。

### Step 3: 确定评估内容中的各字段

综合素材内容，逐项分析并填写以下字段：

| 字段 | 填写方式 | 选项 |
|------|---------|------|
| Trae Session ID | 留空占位，用户手动填写 | |
| User Prompt | 从 `.prompt/previous_prompt.md` 获取 | |
| 当前对话是第几轮 | 检查 `.history/` 目录下该 Task 已有文件的最大轮次数 + 1；无则默认为 1 | |
| 任务类型 | 单选 | Bug修复 / 0-1代码生成 / Feature迭代 / 代码理解 / 代码重构 / 工程化 / 代码测试 |
| 业务领域 | 单选 | 全栈Web应用 / Web前端 / 纯后端服务 / 游戏开发 / 数据分析与可视化 / 3D/交互可视化 / AI/ML应用 / 科学计算 / 命令行工具 / 桌面应用（含GUI）/ 自动化与工具脚本 |
| 修改范围 | 单选 | 无需修改 / 单文件 / 模块内多文件 / 跨模块多文件 / 跨系统多模块 |
| 影响范围 | 单选 | 无影响 / 单模块 / 局部 / 全局 |
| prompt 中是否存在兼容性/单元测试 | 单选 | 是 / 否 |
| 上下文压缩占比 | 留空占位，用户手动填写 | |
| 任务难度 | 单选 | 简单 / 一般 / 困难 / 地狱 |
| 任务是否完成 | 单选 | 完成了任务 / 未完成任务 |
| 产物及过程是否满意 | 单选 | 满意 / 不满意 |
| 不满意原因 | 当"未完成任务"或"不满意"时，必须填写具体原因，尽量多找问题。整体评估中不满意占比应达到 70%。注意：原因内容不能使用任何 markdown 语法（不加粗、不下划线、无列表标记等），纯文本描述，去除 AI 味。 | |

### Step 4: 生成评估文件

在 `.history/` 目录下生成评估文件，内容格式如下：

```markdown
- Trae Session ID: 
- User Prompt: {从 previous_prompt.md 获取的 prompt 内容}
- 当前对话是第几轮: {轮次数}
- 任务类型: {单选值}
- 业务领域: {单选值}
- 修改范围: {单选值}
- 影响范围: {单选值}
- prompt 中是否存在兼容性/单元测试: {是/否}
- 上下文压缩占比: 
- 任务难度: {单选值}
- 任务是否完成: {完成了任务/未完成任务}
- 产物及过程是否满意: {满意/不满意}
- 不满意原因: {当未完成或不满意时填写，否则留空}
```

### Step 5: 输出确认

向用户汇报评估文件已生成，告知文件名和路径，同时提醒用户手动填写 Trae Session ID 和上下文压缩占比两个字段。
