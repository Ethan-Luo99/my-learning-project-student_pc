# Prompt 出题要求

## 基本规则

Prompt 可以是自然语言描述，也可以是长篇幅的 PRD。

若模型未在首轮对话后完成任务，或需要追加需求，专家可以继续与模型交互，但至多不超过 5 轮。

需要单独针对【每一轮对话】标注这一轮的【任务完成度】和【过程满意度】，并说明未完成 / 不满意的原因。对话满五轮后，无论模型是否完成任务，都不应继续引导，应创建新的任务对话，开始新的题目。

### 特别说明

- 对于显然的工程故障（例如网络波动/模型不稳定导致请求失败），可以不计入轮次。
- 但是，对于思考次数超限需要人为"继续"的情况，应计入轮次。

---

## 任务类型

整体任务类型的分布偏序关系如下：0-1代码生成 > Feature迭代 ≈ Bug修复 > 代码理解 ≈ 代码重构 > 其他

| 任务类型 | 说明 |
|---------|------|
| 0-1 代码生成 | 基于 PRD 或自然语言描述的新项目/完整模块构建 |
| Feature 迭代 | 在已有功能基础上的平滑扩展 |
| Bug 修复 | 逻辑修复、报错排查、安全漏洞修补 |
| 代码理解 | 解释逻辑、寻找潜在风险、回答技术细节 |
| 代码重构 | 性能优化、可读性提升、解耦 |
| 工程化 | 环境配置、依赖管理、CI/CD 相关 |
| 代码测试 | 单测、集成测试等 |

---

## 业务领域

整体业务领域的分布偏序关系如下：大前端与服务端类 > 垂直业务类 > 其他

| 业务领域 | 说明 |
|---------|------|
| 大前端与服务端类 | 纯后端 API 服务、Web 前端、全栈 Web 应用 |
| 垂直业务类 | 游戏开发、数据分析与可视化（如 Dash/Streamlit） |
| 前沿技术 | 3D/交互可视化、AI/ML 应用、科学计算 |
| 端侧与基础工具 | 命令行工具、桌面应用（含GUI）、自动化与工具脚本 |

---

## 修改范围

整体修改范围的偏序关系如下：跨模块多文件 > 跨系统多模块 > 模块内多文件 > 单文件 > 无需修改

| 修改范围 | 说明 |
|---------|------|
| 无需修改 | 代码理解等无需代码生成的任务 |
| 单文件 | 局部逻辑修改 |
| 模块内多文件 | 涉及同一模块内的接口定义与实现协同 |
| 跨模块多文件 | 涉及多个逻辑单元的调用链修改 |
| 跨系统多模块 | 涉及前后端联调、跨服务调用、或数据库-后端的垂直改动 |

---

## 任务难度

需根据 repo、技术栈、题目自行判断任务难度。可以主观。

| 难度 | 说明 |
|------|------|
| 简单 | 任务意图明确，主要沿已有模式做机械修改或局部实现；即使改多个文件，只要模式高度一致、核心判断很少，也算简单。 |
| 一般 | 需要理解局部到跨模块的调用/数据流，做一些实现决策；有若干边界条件或约束，但不需要重新设计架构。 |
| 困难 | 需要整合多个模块/系统约束，做关键设计取舍，处理复杂状态、兼容性、权限、并发、性能或异常链路。 |
| 地狱 | 即使 SOTA 模型也大概率失败，通常需求很开放或约束很隐蔽，需要极其深度项目理解、架构级判断、多步复杂的调试验证，或正确性依赖大量边界场景。 |

---

## 质检点

### 1. 整体质量维度

- 数据总量统计
- 整体合格率
- 数据多样性（是否覆盖了各场景，是否出现了雷同的题目）

### 2. 标注合规性

- 轮次合规（是否超5轮、窗口拆分是否正确）
- 字段完整（是否有字段缺失）
- 题目分类准确性（任务类型、业务领域、修改范围）
- 任务完成度判断一致性（是否在原因中提到产物有问题却标注了任务完成，或反之）

### 3. 原因撰写质量

- 语句通顺，表达清晰完整
- 是否出现了极其明显的过程及产物问题却偷懒没有写出
